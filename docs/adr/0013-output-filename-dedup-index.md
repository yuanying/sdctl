# 0013. Output Filename Deduplication Index

- Date: 2026-06-11
- Status: Accepted

## Context

ADR 0012 では、バッチ生成時にバッチカウントを元に連番インデックスを付与する方式を採用していた（例: `xxx.0.png`, `xxx.1.png`）。しかし、この方式では同一出力先に複数回実行した場合にファイルが上書きされてしまう問題がある。保存時にすでに同名ファイルが存在するかどうかを確認し、衝突を回避する方式に変更する。

## Decision

- ファイル保存時に同名ファイルが存在する場合、自動的にインデックスサフィックスを付与して保存する。
- Suffix形式: `<stem>.<NNNN><ext>`（例: `output.0001.png`）
  - 4桁ゼロパディング。9999を超えた場合は桁数を動的に拡張する。
- インデックスの決定方法: 保存先ディレクトリをスキャンし、同一ベース名で既存の最大インデックス + 1 を使用する。
- **単発保存（batch size = 1）**: まず `output.png` を試み、存在する場合は `output.0001.png` から始まるインデックスを使用する。
- **バッチ保存（batch size > 1）**: 最初からインデックス付き（`output.0001.png` 始まり）で保存する。`output.png`（インデックスなし）はバッチでは生成しない。
- このロジックは共通ユーティリティ関数として実装し、`txt2img` / `img2img` の両コマンドから呼び出す。

## Consequences

- 同一出力先に複数回実行してもファイルが上書きされず、すべての生成結果が保持される。
- バッチ生成時のインデックスがバッチサイズではなく既存ファイルに依存するため、実行ごとに連続したインデックスが付与される。
- ADR 0012 のバッチカウントベースの連番方式は廃止される。
- 共通ユーティリティ化により、将来のコマンド追加時も同一ロジックを再利用できる。
